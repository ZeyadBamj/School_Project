## 🧩 كيف يعمل النظام؟

### 1. قاعدة البيانات (Database Layer)

تم بناء كلاس `DataBase` لإدارة الاتصال بـ SQLite.

أهم نقطة هنا:

```python
self.connection.row_factory = sqlite3.Row
```

وهذا يسمح لك بالتعامل مع النتائج كـ dictionary بدل tuple.

---

### 2. الORM (Model Layer)

تم بناء ORM بسيط جدًا يدويًا باستخدام:

- ا`Model` → الكلاس الأساسي
- ا`Field` → لتعريف أنواع الحقول

مثال:

```python
class Students(Model):
    student_id = IntegerField(unique=True, nullable=False)
    first_name = CharField(max_length=30)
```

💡 الفكرة:
بدل ما تكتب SQL يدويًاالنظام يولد الجداول تلقائيًا.

---

### 3.ال Repository Pattern

نمط تصميم هيكلي يُستخدم بشكل أساسي لعزل منطق التعامل مع البيانات (قاعدة البيانات) عن باقي أجزاء التطبيق.

طبقة وسيطة بدل ما تتعامل مباشرة مع قاعدة البيانات

المستودع طبقة فوق طبقةال ORM
ينظم استخدامه بدل ما تستخدم ORM مباشرة في كل مكان


#### اStudentRepository

- اadd
- اupdate
- اdelete
- اexists

#### اLessonRepository

- ربط الطلاب بالدروس
- إدارة العلاقات (Many-to-Many)

الفائدة:

- فصل المسؤوليات
- سهولة التغيير
- سهولة الاختبار
- تنظيم الكود

### 4. الاختبارات (Testing)

تم استخدام:

```
unittest
```

مع قاعدة بيانات:

```
":memory:"
```

هذه نقطة مهمة جدًا:
قاعدة البيانات في الذاكرة تحل مشاكل:

- تضارب البيانات
- بطء الاختبارات
- التلوث بين الاختبارات

وتم انشاء الاختبارات للدوال التي تم انشائها فيما بعد في ملف student_repo.py و lesson_repo.py
وذلك باستخدام مبدأ TDD (Test Driven Develpoment):

🔴 الأحمر (Red): كتابة اختبار فاشل (لأن الميزة لم تُبرمج بعد).

🟢 الأخضر (Green): كتابة أقل قدر من الكود لاجتياز الاختبار.

♻️ إعادة الهيكلة (Refactor): تحسين الكود المكتوب مع التأكد من بقاء الاختبار ناجحاً.

والحفاظ على تنسيقها باستخدام AAA لسهولة القراءة:

1. ا Arrange
المعنى: تجهيز كافة الإعدادات اللازمة للاختبار.

الوظيفة: إنشاء الكائنات (Objects)، تهيئة البيانات، وإعداد السياق الذي سيتم اختبار الكود فيه.

2. ا Act 
المعنى: تنفيذ الوظيفة أو الطريقة (Method) المراد اختبارها.

الوظيفة: تشغيل الجزء الخاص بالكود الذي تريد التأكد من عمله.

3. ا Assert
المعنى: التحقق من أن النتيجة الحقيقية تطابق النتيجة المتوقعة.

الوظيفة: مقارنة النتيجة الفعلية (Actual Result) بما كنت تتوقعه (Expected Result).

### إعادة تصميم البرنامج Refactoring

وذلك بأتباع مبدأ: Rule of Three:

المرة الأولى: قم بكتابة الكود للقيام بالمهمة (المرة الأولى).

المرة الثانية: إذا احتجت لنفس الكود مرة أخرى، قم بنسخه ولصقه (لا بأس بذلك).

المرة الثالثة: إذا احتجت لنفس الكود للمرة الثالثة، يجب عليك إعادة هيكلته (Refactor) واستخراجه في إجراء أو دالة (Function/Method) جديدة.
